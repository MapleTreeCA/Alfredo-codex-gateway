# Gateway 记忆与上下文机制开发总结

## 文档目的
本文档用于记录 `codex-gateway` 当前“上下文记忆 + 记忆检索 + 模型注入”整体链路、实现策略、配置项和阶段进度，便于后续迭代对齐。

## 时间范围
- 主要开发时间：2026-03-10 到 2026-03-11
- 当前基线版本：已支持持久化配置、会话上下文窗口、跨会话时窗检索、短记忆注入

## 当前整体流程
1. 用户请求进入 `/api/chat`。
2. 网关读取运行时配置：
- `model`
- `effort`
- `verbosity`
- `online`
- `context_messages`
- `memory_recall_days`
3. Codex 客户端构建请求：
- 输入消息：当前会话最近 `context_messages` 条 + 当前用户输入
- 指令注入：基础系统提示 + 运行模型标识 + 记忆提示（可选）
4. 若命中记忆检索条件，则执行“最近 N 天跨会话检索”，抽取简短提示文本拼入 `instructions`。
5. 请求发送到 Codex API，返回结果后写入 SQLite 记忆库。

## 关键配置项
`context_messages`
- 含义：每轮发送给模型的当前会话历史条数上限。
- 作用：控制短期对话连续性。
- 范围：`1..200`。
- 默认：`10`。

`memory_recall_days`
- 含义：检索“相关长期记忆”时回看天数。
- 作用：控制长期记忆召回时间窗。
- 范围：`1..365`。
- 默认：`30`。

配置持久化策略：
- 保存位置：SQLite `runtime_settings` 表（`runtime_config` JSON）。
- 启动行为：服务启动时自动加载并生效。
- 修改行为：前端保存后即时生效并持久化。

## 记忆检索与压缩注入策略
检索范围：
- 记忆库全局检索（不限定 session）。
- 时间过滤：`created_at >= now - memory_recall_days`。

触发条件：
- 用户输入长度至少 3 个 rune。
- 记忆库可用且配置天数有效。

注入限流（避免 token 暴涨）：
- 候选扫描上限：24 条。
- 注入条数上限：`min(3, context_messages / 4)`。
- 单条截断上限：110 字符。
- 注入总长度上限：280 字符。
- 去重：与当前会话历史和当前问题文本去重。

## 现阶段进度
已完成：
- 网关内建 OAuth 登录、回调、状态检查。
- Web 控制台可配置模型/推理强度/verbosity/上下文条数/检索天数/TTS 参数。
- Runtime 配置 SQLite 持久化（启动自动恢复）。
- 会话记忆写入 SQLite、记忆搜索分页。
- 对话调用支持“近期上下文 + 短记忆注入”双路径。
- 页面状态灯改造（成功/警告/失败状态悬浮提示）。

进行中：
- 记忆召回策略参数可视化调优（候选量、截断策略、权重排序）。

待完成：
- 将“召回条数、单条长度、总注入长度”开放为可配置项。
- 增加召回命中可观测性（每轮命中条数、截断前后长度、来源会话）。
- 增加“按主题/标签”聚合记忆视图（辅助人工验证）。

## 风险与注意点
- `context_messages` 过大可能显著提高输入 token。
- `memory_recall_days` 过大可能引入过时语义，需要后续做时效权重。
- 目前为关键词检索，不是向量语义检索；复杂语义召回能力有限。

## 建议默认策略
- 日常模式：`context_messages=10`，`memory_recall_days=30`。
- 成本敏感模式：`context_messages=6`，`memory_recall_days=14`。
- 强记忆模式：`context_messages=16`，`memory_recall_days=60`（需观察 token）。
